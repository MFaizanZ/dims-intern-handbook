Software & Web 101

Before we can explain how DIMS is built, you need the vocabulary that every software person uses without thinking. This chapter starts from absolute zero — what software even is — and ends with you understanding the words you'll hear in every standup. Each idea closes with “…and that's why DIMS uses X.”

What software actually is

Software is just a set of written instructions a computer follows, one after another. You write those instructions in a programming language — a strict, unambiguous way of telling the machine exactly what to do (computers do not guess or interpret intent; a missing comma can break everything).

Computers only understand raw 1s and 0s, so something has to translate your human-readable code into machine instructions. There are two styles: compiled languages translate the whole program up front into a machine file you then run; interpreted languages are translated line by line as they run. That's the one-line version — you don't need more.

Two languages matter at CNS. Python runs the backend and all the AI — it's readable and is the universal language of machine learning. JavaScript (and its stricter cousin TypeScript, which adds type-checking to catch mistakes early) runs everything you see in the browser. …and that's why DIMS is Python on the back and TypeScript on the front.

🧩 Analogy

A programming language is like a recipe written for a very literal cook who has zero common sense. If the recipe says “add salt” but never says how much, the cook adds nothing — or the whole jar. Code is that recipe: every step must be spelled out exactly, in the right order.

The single most important model: client vs server

If you remember one thing from this chapter, make it this. Almost every web system — DIMS included — is split into two halves that live in different places and talk over the internet:

  • The frontend (a.k.a. the client) — the part that runs on your screen, in your web browser. It's what the user sees and clicks.
  • The backend (a.k.a. the server) — the part that runs on a powerful computer somewhere else, doing the heavy work: storing data, running AI, talking to the drone.
🔑 The one idea to remember

Frontend = what's on the user's screen. Backend = the machine doing the work elsewhere. They are separate programs that send messages to each other. Nearly every “how does X work?” question is answered by asking “is that the frontend or the backend?”

🧩 Analogy — the restaurant

Picture a restaurant. The dining room is the frontend: the menu, the tables, where you (the user) sit and place orders. The kitchen is the backend: you never go in there, but it does the real cooking. The pantry is the database: where all the ingredients are stored. A waiter carries orders from the dining room to the kitchen and brings food back — that waiter is the API (next section). You can't taste a dish from the dining room; you have to ask the kitchen. That's exactly how the browser asks the server for data.

In DIMS, the frontend is built with Next.js and React and runs in the operator's browser; the backend is FastAPI, written in Python. The screenshot below is the dining room — the frontend the user sees — while a separate Python kitchen feeds it data.

This page is an HTML file styled by a CSS file, with a little JavaScript filling in the sidebar and the read-time pill. …and that's why the DIMS console — being a web page — is ultimately HTML, CSS and JavaScript too, just generated by a framework instead of typed by hand.

Frameworks & libraries — why you don't build from scratch

Writing every button, every network call, every 3D effect from raw JavaScript would take forever and be full of bugs. So developers stand on the shoulders of others by using:

  • A library — a pack of pre-written code you call when you need it (you stay in charge).
  • A framework — a bigger structure that runs your code and gives the whole app a shape (it's in charge; you fill in the blanks).

DIMS uses React (a library for building the UI out of reusable components — think Lego bricks like “a button” or “an alert card”), wrapped in Next.js (a full web-app framework that handles pages, routing and serving). For the 3D digital twin it uses CesiumJS, a library that renders real-world geographic terrain in the browser — we'll go deep on it in chapter 6 and chapter 14. …and that's why DIMS could build a flight console in months, not years.

How frontend and backend talk: APIs

The frontend can't reach into the backend's memory — they're separate programs. They communicate by sending messages over the internet through an API (Application Programming Interface): an agreed list of requests the backend will answer. The frontend sends a request; the backend sends back a response.

DIMS's API follows the REST style — the standard recipe for web APIs. Each request is made over HTTP (the web's transport protocol) using a verb that states intent:

  • GET — “give me data” (e.g. fetch the list of missions).
  • POST — “here's new data, create/do something” (e.g. start a flight).

The data itself travels as JSON — a simple, human-readable text format of keys and values that both Python and JavaScript understand. Here's a tiny REST exchange, the kind the DIMS console makes constantly:

GET /api/missions/42        → the frontend asks the backend

200 OK                      ← the backend answers
{
  "id": 42,
  "name": "Bomunsan patrol",
  "status": "flying",
  "battery_pct": 87
}

One catch: a plain request/response is a one-time question. But the operator needs the drone's position to update live, many times a second, without re-asking. For that DIMS uses SSE (Server-Sent Events) — a kept-open connection where the backend pushes new data to the browser the moment it happens. …and that's why a new fire alert appears on your console instantly, with no refresh.

Databases — where the data lives

The backend has to remember things between requests: missions, flight telemetry, alerts, registered devices. That permanent store is a database. DIMS uses a relational database, which organises data into tables of rows and columns (like a set of linked spreadsheets) — a “missions” table, an “alerts” table, and so on, with relationships between them.

The specific one is PostgreSQL (“Postgres”), a battle-tested open-source relational database. Every mission you plan, every coordinate of telemetry, every alert raised, and every drone registered is a row in Postgres. …and that's why a flight you ran yesterday is still there to replay today.

The developer's toolbox

You'll see these tools on every engineer's screen. You don't have to master them yet — just recognise them.

The terminal (CLI)

A terminal (or command-line interface, CLI) is a text window where you type commands instead of clicking. It looks intimidating but it's just a faster way to tell the computer to do things — start a server, run a script, move files.

Git & version control

Git is the time machine for code. It records every change as a commit, lets many people work on the same project without overwriting each other, and lets you roll back mistakes. A project tracked by Git is called a repository (“repo”). As you saw in chapter 1, DIMS has two repos: dims-platform (the real product) and dims-showcase (the prototype site). A typical Git moment:

git add .
git commit -m "Add live fire alert to console"
git push

Package managers

Remember those libraries and frameworks? A package manager downloads and installs them for you, with the correct versions. JavaScript uses npm; Python uses pip (DIMS uses a fast modern variant called uv). One command pulls in hundreds of building blocks. …and that's why React or CesiumJS land in the project without anyone copying files by hand.

Running a dev server

While building, a developer “runs a dev server” — they start the app locally on their own machine so they can see changes instantly in a browser at an address like localhost:3000. It's the same app the world will use, just running privately for testing. The DIMS backend and frontend each run as their own dev server during development.

Frontend vs backend — who does what

These two halves have totally different jobs. Flip between the tabs to lock it in.

Runs in: the user's web browser.

Built with: React + Next.js (TypeScript), plus CesiumJS for 3D.

Job: draw the console, show the 3D twin and video, capture clicks, and ask the backend for data via the API / SSE.

Does NOT: store data permanently or talk to the drone directly.

Runs on: a server machine (the DIMS cloud).

Built with: FastAPI (Python).

Job: the heavy lifting — talk to the dock & drone, run AI on the video, raycast detections into coordinates, read/write Postgres, and answer the frontend's API requests.

Does NOT: render the screen — it only sends data.

Want the deep version? Where does the AI run, frontend or backend?

The computer-vision AI that boxes fire and smoke runs on the backend, on a GPU, because it's heavy number-crunching that a browser can't do well. The backend takes the drone's video, runs the model, gets the boxes, turns them into coordinates, and then sends only the results (boxes + an alert) to the frontend over the API/SSE. So the impressive AI you see drawn on screen was actually computed in the Python kitchen and merely displayed in the dining room. More in chapter 15.

“Software is instructions in a programming language; DIMS uses Python for the backend/AI and TypeScript for the frontend. Every web app splits into a frontend (runs in your browser — DIMS uses React + Next.js) and a backend (runs on a server — DIMS uses FastAPI in Python), like a restaurant's dining room and kitchen. They talk through a REST API over HTTP using JSON, and the backend pushes live updates via SSE. Data lives in a PostgreSQL database. Developers build with the terminal, Git (DIMS has two repos), package managers like npm and pip/uv, and by running a local dev server.”
